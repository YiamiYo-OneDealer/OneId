# Decision Log — OneId Product Brief

## 2026-05-21

### Scope: POC only
Purpose of the brief is to support a feasibility POC, not a production launch. Brief right-sized accordingly — no investor-grade rigor, no production SLA design.

### Platform: OpenIddict on .NET
Chosen for .NET-native alignment with OneDealer's stack and team expertise. OpenIddict provides standards-compliant OAuth2/OIDC without requiring a polyglot infrastructure.

### Why not off-the-shelf (Keycloak, Auth0, Azure AD B2C, Okta)
Validated during discovery: the combination of fine-grained permissions + multi-dimensional ABAC + row-level data filtering + licensing integration would require deep customization of any standard IDP, rivaling the effort of building purpose-fit on OpenIddict.

### IDP chaining for existing integrations
SAML2 and Okta were added incrementally to the current system. OneId will use OpenIddict's IDP chaining support to preserve these integrations rather than replacing or re-implementing them.

### Licensing integrated into OneId (not a separate service)
Licensing state (seat count today, concurrent and usage-based planned) lives in OneId and is emitted as JWT claims. OneDealer consumes claims at runtime without a separate licensing service call. Tradeoff accepted: OneId carries more responsibility than a pure IDP, but the runtime simplicity and single-ownership model were preferred.

### Management UI is the primary UI concern
Authentication UI (login, MFA, password reset) is considered relatively straightforward. Management UI for the full entity model is the primary UI challenge and investment. Frontend: React (confirmed).

### Simple MFA included at POC stage
Confirmed in scope. Complexity to be determined during POC.

### Tenant self-service administration confirmed
Tenant administrators can manage their own users, groups, roles, and dimensional assignments within their tenant scope. Scoped management view enforces tenant isolation.

### Authorization model mapped to industry-standard terminology
Internal names reconciled to standard terms:
- "Dimensions" (Company/Location/Branch/Make/MarketSegment) → ABAC attributes
- "Roles / RoleGroups" → RBAC roles with hierarchical grouping
- "UserGroups" → Groups (standard RBAC groups)
- "BusinessCaseIDs" → Permissions (fine-grained, function-level)
- "Teams" → Deferred from POC; likely an ABAC dimension, to be confirmed post-POC
- "Tenants" → Tenants (standard multi-tenancy)

### Dimensions confirmed as row-level data security via JWT claims
Dimensions are not only feature-level gates — they drive row-level query filtering across all entities in the OneDealer database. The IDP must embed dimensional attributes accurately in every issued token. OneDealer applies these claims as query filters. This is a first-class architectural concern, not a secondary attribute.

### Permission naming: numeric IDs → string identifiers
Current system uses numeric BusinessCaseIDs. Decision to transition to string-identified permissions (e.g., `crm.invoice.create`, `erp.pricing.discount.approve`) for JWT readability and alignment with OAuth2/OIDC conventions. Production will map existing numeric IDs one-to-one to string identifiers when ready. POC uses generic representative strings.

### End customers out of scope
Car buyers / end consumers explicitly excluded from OneId scope for OneDealer v2.

### Scale
Few hundred tenants × few hundred to ~1,000 users per tenant. Not called out as a primary concern at POC stage; to revisit if full build proceeds.
